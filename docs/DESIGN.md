# 设计决策记录

本文档记录 Reasonix MCP Server 中的关键设计决策及其背后的思考过程。

---

## 决策 1：为什么选择 stdio 作为传输层？

**背景**：MCP 协议支持多种传输层，包括 stdio、SSE、WebSocket 等。

**选择**：stdio (JSON-RPC 2.0 over stdin/stdout)

**理由**：
1. **部署零配置**：不需要端口、不需要网络暴露、不需要 TLS 证书。宿主 Agent 直接 `spawn` 一个子进程即可。
2. **安全边界清晰**：子进程的天然沙箱 — 即使 Server 被攻破，也只是一个普通用户进程。
3. **生命周期绑定**：Server 进程跟随宿主 Agent 一起启动和退出，不需要额外的守护进程管理。
4. **协议简单**：按行分隔的 JSON，没有粘包问题，不需要 framing 协议。

**代价**：无法远程部署 Server，必须和宿主 Agent 在同一台机器上。考虑到编码任务需要读写本地文件系统，这反而是合理的约束。

## 决策 2：为什么每个任务一个独立进程？

**背景**：可以实现为单进程多线程，或单进程异步事件循环。

**选择**：每个任务一个独立子进程（Worker）

**理由**：
1. **崩溃隔离**：Worker 崩溃不会拉垮 Server 和其他任务。这是最重要的。
2. **资源隔离**：每个 Worker 有自己的内存空间，一个任务的内存泄漏不会影响其他任务。
3. **信号处理简单**：杀掉一个 Worker 就是 `kill(pid)`，不需要在线程模型中处理复杂的取消 token。
4. **调试友好**：可以单独启动一个 Worker 进程，用 `node --inspect` 调试。

**代价**：进程创建有一定开销（几十毫秒），但对于通常持续数十秒到数分钟的编码任务来说完全可忽略。

## 决策 3：为什么零运行时依赖？

**背景**：可以用 `@modelcontextprotocol/sdk`、toml 解析库、glob 库等来加速开发。

**选择**：不使用任何 npm 依赖

**理由**：
1. **供应链安全**：每个 npm 依赖都是一个潜在的攻击面。2024 年 xz 后门事件和各类恶意 package 事件证明了这个风险是真实的。
2. **故障面控制**：依赖越少，不可控的崩溃点越少。
3. **部署简单**：不需要 `npm install`，直接把仓库 clone 下来就能跑。
4. **足够简单**：我们的需求（JSON-RPC 解析、TOML 子集解析、简单 glob、文件操作）用原生 API 完全能覆盖。

**代价**：需要自己实现一些轮子（TOML 解析、glob）。但考虑到代码量只有几百行，这个代价是值得的。

## 决策 4：为什么用 SEARCH/REPLACE 而不是行号编辑？

**背景**：AI 编辑文件有两种主流方式：基于行号的 patch，和基于文本匹配的 SEARCH/REPLACE。

**选择**：SEARCH/REPLACE，且要求 SEARCH 文本在文件中唯一出现

**理由**：
1. **行号不稳定**：AI 在多次编辑后很难准确跟踪行号变化，导致 patch 偏移。
2. **文本匹配更可靠**：即使文件内容被其他编辑修改过，只要 SEARCH 文本仍然唯一存在，就能正确替换。
3. **唯一性校验**：如果 SEARCH 出现多次，说明 AI 提供的上下文不够精确，拒绝执行可以防止误替换。
4. **原子性**：`edit_file` 先检查唯一性，再写入。如果检查通过但写入时文件被外部修改（竞态），虽然理论上可能出错，但在单进程 Worker 模型中不会发生。

**回退策略**：如果 SEARCH 不唯一，AI 可以通过扩大 SEARCH 范围（多包含上下文行）来让它变得唯一，或者改用 `write_file` 重写整个文件。

## 决策 5：为什么对话历史要完整持久化？

**背景**：Worker 的执行过程是 AI 和工具的多轮对话。每次 API 调用都要把完整的 conversation 传上去。

**选择**：每轮对话后都保存完整的 messages 数组到状态文件

**理由**：
1. **续跑支持**：这是实现 `reasonix_resume_task` 的基础。没有历史，续跑就等于重头来过。
2. **调试支持**：可以从状态文件中完整还原一次任务的执行过程，排查问题。
3. **成本可接受**：对话历史通常几十到几百 KB，现代 SSD 可以轻松处理。

**优化点**：历史中的 tool_calls 需要做深拷贝后再保存，防止引用共享导致的序列化问题。

## 决策 6：为什么 review 模式是同步阻塞的？

**背景**：`reasonix_review_changes` 内部也会 spawn 一个 Worker 去做审查。

**选择**：Server 等待 Worker 完成后才返回结果（同步阻塞，5 分钟超时）

**理由**：
1. **语义匹配**：宿主 Agent 调用 review 时期望立即得到 ALLOW/BLOCK 的 verdict，然后决定下一步行动。如果异步返回，宿主需要额外的轮询逻辑，体验割裂。
2. **审查通常很快**：代码审查通常只需要几轮到十几轮对话，远小于复杂编码任务。
3. **超时保护**：5 分钟超时后自动返回 ALLOW，防止无限阻塞。

## 决策 7：为什么 delegate_task 是同步阻塞的？

**背景**：子任务委派可以让 Worker 把独立子问题拆给其他 Worker。

**选择**：`spawnSync` 阻塞式调用，父 Worker 等待子 Worker 完成

**理由**：
1. **简化状态机**：如果子任务是异步的，父 Worker 需要在等待期间保持"挂起"状态，同时处理其他消息，这会让 Worker 的事件循环复杂化。
2. **DeepSeek API 的限制**：当前 API 不支持真正的并行工具执行。即使子任务异步跑，父 Worker 也不能在等的同时做其他事。
3. **足够简单**：子任务通常是小范围的问题，执行时间可控。

**未来方向**：如果需要真正的并行子任务，可以改为异步 spawn + 状态轮询，但会显著增加复杂度。

## 决策 8：为什么 TOML 解析器是自写的？

**背景**：配置格式选择。可选方案：JSON、YAML、TOML、INI、纯环境变量。

**选择**：TOML（手写解析器）

**理由**：
1. **TOML 比 JSON 更适合人类手写**：支持注释、不需要引号包围键名、段落清晰。
2. **TOML 比 YAML 更安全**：YAML 的隐式类型转换和 `!!python/object` 等标签是已知的安全风险。
3. **自写解析器足够覆盖需求**：我们只需要处理 `[section]`、`key = value`、字符串/数字/boolean 三种值类型。完整 TOML 规范很复杂，但我们用不到。
4. **保持零依赖**：不引入 `@iarna/toml` 等库。
